# Chinmina Bridge

[![Chinmina Docs](https://img.shields.io/badge/docs-chinmina-dev?style=plastic&label=docs&link=https%3A%2F%2Fdocs.chinmina.dev)](https://docs.chinmina.dev)
[![Ask DeepWiki](https://deepwiki.com/badge.svg)](https://deepwiki.com/chinmina/chinmina-bridge)

**Connect Buildkite to GitHub with secure, short-lived tokens.**

Chinmina Bridge allows Buildkite agents to securely generate GitHub API tokens
that can be used to perform Git or other GitHub API actions. It is intended to
be an alternative to the use of SSH deploy keys or long-lived Personal Access
Tokens.

![High level Chinmina diagram](docs/chinmina-high-level.png)

The bridge itself is an HTTP endpoint that uses a [GitHub
application][github-app] to create [ephemeral GitHub access
tokens][github-app-tokens]. Requests are authorized with a [Buildkite
OIDC][buildkite-oidc] token, allowing a token to be created just for the
repository associated with an executing pipeline.

## Installation

Chinmina Bridge is distributed as a multi-platform Docker image (Linux
amd64/arm64) and as pre-built binary archives on [GitHub
Releases](https://github.com/chinmina/chinmina-bridge/releases). Every release
artifact carries a build-provenance attestation and is signed with Sigstore
`cosign` — see [Verifying releases](#verifying-releases).

<details>
<summary><strong>Docker (recommended)</strong></summary>

The preferred way to run Chinmina Bridge is via its [Docker
image](https://hub.docker.com/r/chinmina/chinmina-bridge), which supports Linux
amd64 and arm64:

```sh
docker pull chinmina/chinmina-bridge:<tag>
```

</details>

<details>
<summary><strong>Manual download</strong></summary>

Download the archive for your platform from the [releases
page](https://github.com/chinmina/chinmina-bridge/releases), verify its
provenance, and put the binary on your `PATH`:

```sh
ARCH=amd64   # or arm64
curl -fsSLO "https://github.com/chinmina/chinmina-bridge/releases/latest/download/chinmina-bridge_linux_${ARCH}.tar.gz"
gh attestation verify "chinmina-bridge_linux_${ARCH}.tar.gz" --repo chinmina/chinmina-bridge
tar -xzf "chinmina-bridge_linux_${ARCH}.tar.gz" chinmina-bridge
install -m 0755 chinmina-bridge ~/.local/bin/
```

</details>

## Verifying releases

Release artifacts carry a [build-provenance
attestation](https://docs.github.com/en/actions/security-for-github-actions/using-artifact-attestations/using-artifact-attestations-to-establish-provenance-for-builds)
(SLSA) generated by the release workflow with
[Sigstore](https://www.sigstore.dev/) keyless signing — there is no long-lived
signing key. Each artifact is bound, by digest, to the source commit and the
workflow that produced it.

**Binary archives.** Install the [GitHub CLI](https://cli.github.com/) (≥ 2.49.0)
and verify a downloaded archive:

```sh
# e.g. ARTIFACT=chinmina-bridge_linux_amd64.tar.gz
gh attestation verify "$ARTIFACT" --repo chinmina/chinmina-bridge
```

Each archive also embeds a `cosign` bundle (`chinmina-bridge.cosign.bundle`) for
`cosign verify-blob`, and `checksums.txt` is published for `sha256sum --check
checksums.txt`.

**Docker image.** The published image is signed keylessly with `cosign`. See the
[release verification guide][docs-verify] for the current `cosign verify` command
and signing identity.

[docs-verify]: https://docs.chinmina.dev/guides/verifying-releases/
